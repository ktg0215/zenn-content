> 第3章でリスティングを最適化し、ストアページの転換率を改善しました。しかし最適化の効果が及ぶのは、ストアページを閲覧した人だけです。日本語のみで運用している拡張は、CWSの検索トラフィックの80%以上をそもそも取り込めていません。この章では「翻訳」ではなく「文化・UX・決済・サポートまで含むローカライゼーション戦略」を解説します。

---

## なぜローカライゼーションが成長レバーなのか

Chrome Web StoreはGoogleが運営しているにもかかわらず、言語ごとに独立した検索インデックスを持っています。日本語で公開した拡張は、英語圏の検索結果にほぼ表示されません。

### 日本語オンリーの天井

日本のChrome拡張ユーザー数は、全CWSユーザーの推定5〜8%です。仮に日本語市場で上位に表示されていたとしても、潜在ユーザーの90%以上には届いていません。

### 英語オンリーの天井

逆も然りで、英語のみで公開すると「日本語で検索するユーザー」を取りこぼします。特に日本語特有の操作（縦書きPDF、日本の行政サイト操作、和文フォント管理）を扱う拡張は、日本語市場の需要を取り込めないまま埋もれます。

### 両対応で市場はどう変わるか

JFF（Japanese Font Finder）で実際に観測したデータです。

| 対応言語 | 月間インストール | インストール構成比 |
|---------|-------------|---------------|
| 日本語のみ（公開直後） | 23/月 | JP 100% |
| 英語追加（3ヶ月後） | 67/月 | JP 41% / EN 59% |
| 8言語対応（6ヶ月後） | 142/月 | JP 28% / EN 48% / その他 24% |

英語を追加しただけでインストール数が約3倍になりました。さらに多言語対応でCWSの検索インデックスが増え、ロングテールのキーワードでも発見されるようになります。

```
18本の拡張 × 8言語 = 144の検索可能なリスティング
（同じ拡張が英語・日本語・中国語・ドイツ語等で別々に検索されるため）
```

## 翻訳 vs ローカライゼーションの違い

`messages.json` のテキストを英語に訳すことは「翻訳」です。「ローカライゼーション」はその先にあります。

| 項目 | 翻訳（Translation） | ローカライゼーション（L10n） |
|-----|-------------------|----------------------|
| UI テキスト | ○ | ○ |
| 日付・数値フォーマット | △ | ○（¥1,200 vs $9.99） |
| UI レイアウト（文字幅対応） | ✗ | ○ |
| 文化的文脈（アイコン・比喩） | ✗ | ○ |
| 決済通貨・価格感 | ✗ | ○ |
| サポート言語 | ✗ | ○ |
| CWSリスティング（キーワード戦略） | ✗ | ○ |

翻訳だけでは「読める」が「刺さらない」コンテンツになります。この章では両方を扱います。

## `_locales/` ディレクトリ構造とi18n実装

### 基本ディレクトリ構造

```
my-extension/
├── manifest.json
├── _locales/
│   ├── en/
│   │   └── messages.json
│   ├── ja/
│   │   └── messages.json
│   ├── zh_TW/
│   │   └── messages.json
│   └── de/
│       └── messages.json
└── ...
```

`manifest.json` に `default_locale` を設定します。

```json
// manifest.json
{
  "manifest_version": 3,
  "name": "__MSG_extensionName__",
  "description": "__MSG_extensionDescription__",
  "default_locale": "en",
  "action": {
    "default_title": "__MSG_actionTitle__"
  }
}
```

### messages.json の書き方

```json
// _locales/en/messages.json
{
  "extensionName": {
    "message": "ReadMark — Reading Position Saver for Long Articles"
  },
  "extensionDescription": {
    "message": "Stop losing your place in long articles. ReadMark automatically saves your scroll position and restores it when you return — no login required."
  },
  "actionTitle": {
    "message": "Save reading position"
  },
  "savedPosition": {
    "message": "Position saved at $PERCENT$%",
    "placeholders": {
      "percent": {
        "content": "$1",
        "example": "73"
      }
    }
  },
  "restoreButton": {
    "message": "Continue reading"
  }
}
```

```json
// _locales/ja/messages.json
{
  "extensionName": {
    "message": "ReadMark — 長文記事のスクロール位置を保存"
  },
  "extensionDescription": {
    "message": "長い記事を途中で閉じても、次に開いたとき続きから読めます。ログイン不要、自動保存。"
  },
  "actionTitle": {
    "message": "読書位置を保存"
  },
  "savedPosition": {
    "message": "$PERCENT$% 地点を保存しました",
    "placeholders": {
      "percent": {
        "content": "$1",
        "example": "73"
      }
    }
  },
  "restoreButton": {
    "message": "続きを読む"
  }
}
```

コード側での呼び出しは統一されます。

```typescript
// 言語を意識せずに使える
const label = chrome.i18n.getMessage('restoreButton');
// en環境 → "Continue reading"
// ja環境 → "続きを読む"
```

### 対応すべき言語の優先順位

全言語に対応するのは非現実的です。ROIの高い順に対応します。

| 優先度 | 言語コード | 理由 |
|-------|---------|-----|
| 1位 | `en` | CWSの最大市場（40%+） |
| 2位 | `ja` | 高いPro転換率（日本人は課金に慣れている） |
| 3位 | `zh_TW` | 台湾・香港・海外中国語圏の需要 |
| 4位 | `de` | ドイツ語圏のPrivacy-first意識が高く課金率も高い |
| 5位 | `fr` | フランス語圏のユーザー数 |
| 6位〜 | `es`, `pt`, `ko` | 費用対効果を見ながら追加 |

## UI設計の文化差：日英で変わるレイアウト

翻訳テキストをそのまま置き換えると、UIが崩れます。文化によってテキスト長が大きく異なるからです。

### テキスト長の違い

同じ意味でも、言語によって文字数が変わります。

| 英語 | 日本語 | 比率 |
|-----|-------|-----|
| Save | 保存 | 英語の方が短い |
| Continue reading | 続きを読む | ほぼ同じ |
| Settings | 設定 | 日本語の方が短い |
| Reading position saved at 73% | 73% 地点を保存しました | 日本語の方が長い |

**対応策**: ボタン幅を固定せず、`min-width` + `max-width` でテキストに応じて伸縮させます。

```css
/* ❌ 固定幅（英語のみで設計したとき） */
.save-button { width: 80px; }

/* ✅ 伸縮対応 */
.save-button {
  min-width: 80px;
  max-width: 160px;
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

### ボタンラベルの文化差

英語と日本語では、同じ機能でも表現が異なります。

| 機能 | 英語（短く命令形） | 日本語（丁寧形） |
|-----|---------------|------------|
| 保存 | Save | 保存する |
| 削除 | Delete | 削除する |
| 同意 | Agree | 同意して続ける |
| Pro版購入 | Upgrade to Pro | Pro版にアップグレードする |

日本語UIで英語式の短い命令形を使うと「粗雑」に感じられることがあります。逆に英語UIで「です・ます調」相当の丁寧表現を使うと冗長に映ります。

## CWSリスティングのローカライズ

CWSのストアページ（タイトル・説明文・スクリーンショット）は言語別に設定できます。これはCWSの多言語機能で、Developer Dashboardから各言語の説明文を設定します。

### キーワード戦略の言語差

**重要**: 英語と日本語では、同じ機能を指すキーワードが異なります。

| 機能 | 英語キーワード | 日本語キーワード |
|-----|------------|--------------|
| スクロール位置保存 | scroll position saver, reading bookmark | スクロール位置 保存, 読書位置 |
| データ抽出 | web scraper, data extraction | スクレイピング, データ抽出 |
| 手順書作成 | procedure documentation, step-by-step guide | 手順書 作成, ステップガイド |

日本語リスティングは日本語のキーワードで最適化し、英語リスティングは英語のキーワードで最適化します。単純な翻訳ではなく、**各言語のユーザーが実際に検索するワード**を調査して設定します。

### スクリーンショットの言語対応

スクリーンショットにテキストオーバーレイを入れている場合、言語別に別画像を用意します。これはCWSのDeveloper Dashboardで言語別に画像を設定できます。

## 価格戦略のローカライズ

ExtensionPay（Stripe連携）を使っている場合、価格設定は通貨を意識する必要があります。

### 通貨の心理的境界

日本人ユーザーにとって「$3.99」は計算が必要です。ドル建て表示が日本円に自動変換される場合でも、「¥600」と「¥597」では印象が変わります。

Stripeでの推奨価格設定（S-Hubでの経験値）：

| ターゲット | 推奨価格 | 理由 |
|---------|--------|-----|
| 英語圏（月額） | $3.99 | $4以下の心理的障壁 |
| 英語圏（年額） | $29.99 | $30未満 |
| 日本（月額） | ¥600 | Kindle Unlimitedと同じ感覚 |
| 日本（年額） | ¥4,800 | 月600×8ヶ月分（2ヶ月お得） |

**ExtensionPayの仕様**: 価格はUSD建てで設定し、Stripeが自動で現地通貨に変換します。ただし変換レートは一定ではないため、`¥600` を厳密に実現したい場合は JPY 建ての商品を別途作成する必要があります。

### 日本市場の課金特性

日本人ユーザーはサブスクリプションへの抵抗感が比較的低い一方、**初期決済時のハードル（クレジットカード入力）が高い**という特性があります。

対応策：
- 「14日間無料トライアル」を明示してクレジットカード入力を正当化
- 日本語で「いつでもキャンセル可能」を注意書き追加
- プライバシーポリシーとサポート連絡先を日本語で用意

## サポート言語の現実

多言語対応すると、多言語のサポート問い合わせが来ます。

### JFFで起きたこと

JFFは日本語ユーザー向けに作った拡張ですが、英語対応後に英語のレビューと問い合わせが増えました。AIを使った対応フローを構築したことで、1日5件程度の問い合わせを30分以内に処理できています。

```
ユーザーからの問い合わせ（任意の言語）
    ↓
Claude / GPT-4に「この問い合わせの言語を検出して日本語で要約して」
    ↓
問題の種類を判断（バグ / 使い方 / 課金 / 機能リクエスト）
    ↓
テンプレート回答を選択 → AIで相手の言語に翻訳
    ↓
送信（実際に読んで微調整してから送ることを推奨）
```

**サポートテンプレートの準備**: よくある問い合わせ5〜10パターンを日本語で作成し、AIに翻訳させます。定型文として使いまわすことで、サポート時間を大幅に削減できます。

## S-Hubの実例：両刀の効果

### JFF（Japanese Font Finder）

日本語話者が「Webページで使われているフォントを一瞬で特定する」ために作った拡張です。英語圏では「デザイナーがWebサイトのフォントを調べる」用途で使われることを想定していませんでしたが、英語対応後に英語圏のデザイナーからのインストールが急増しました。

日本語フォントの多様さを求めるユーザーだけでなく、「Webフォント検出ツール」として汎用的に使われていることがわかりました。**ローカライズすることで、作り手が想定していなかったユースケースを発見できる**ことがあります。

### ReadMark

英語圏メインで設計した拡張ですが、日本語対応を追加したことで日本の技術ドキュメントを読む開発者層からのインストールが増加しました。英語・日本語の両リスティングを最適化した結果、月間インストールの28%が日本語圏からになっています。

:::
