title: "第3章 — リスティング最適化：見つけた人をインストールに変える"
---

> 第2章でCWS SEOを最適化し、検索で見つけてもらえるようになりました。しかし「見つけてもらうこと」と「インストールしてもらうこと」は別のゲームです。ストアページは実質的にランディングページです。この章では、転換漏斗のどこで離脱が起きているかを把握し、インストール率（CVR）を改善する手順を解説します。

---

## 転換漏斗：どこで離脱しているか

CWSの転換漏斗を意識したことがありますか。ユーザーが拡張をインストールするまでには、少なくとも3つの関門があります。

```
検索結果に表示 → クリック（CTR）→ ストアページ閲覧 → インストール（CVR）
    ↑ 第2章で最適化            ↑ ここが今章のターゲット
```

第2章のタイトル最適化でCTR（クリック率）を改善しました。しかし、クリックされたあとのCVR（インストール転換率）を改善しなければ、増えたクリックが無駄になります。

### CVRの実測値（Procshot事例）

Procshotでは、CWSのデータから以下が観測できました。

| 期間 | インプレッション | クリック（CTR） | インストール | CVR |
|-----|-------------|------------|---------|-----|
| SEO最適化前 | 240/週 | 5（2.1%） | 11/週 | 46% |
| タイトル変更後 | 1,056/週 | 40（3.8%） | 47/週 | **33%** |
| リスティング最適化後 | 1,056/週 | 40（3.8%） | 64/週 | **53%** |

タイトル変更だけだとCVRが実は下がっていた（より広いキーワードで引き付けるため、ミスマッチが増える）。リスティングを最適化してはじめてCVRが53%まで回復・向上しました。

**ReadMarkでは**: 一行目のhook文を変更しただけでCVRが73%→89%に改善した事例もあります。詳細は「ヒーロー情報の設計」で説明します。

---

## CVRに影響する要素を特定する

ストアページのどこをユーザーが見ているかは、CWSのAnalytics（Chrome Developer Dashboard）で確認できます。ただし、詳細なヒートマップは取れないため、構造的な仮説から始めます。

私が18本を運用して確認した「CVRへの影響度」の推定値：

| 要素 | CVRへの影響度（推定） | 変更コスト |
|-----|-----------------|---------|
| 1枚目スクリーンショット | ★★★★★ | 高（デザイン必要） |
| Short descriptionの1行目 | ★★★★★ | 低（テキスト変更のみ） |
| タイトル（ベネフィット表現） | ★★★★☆ | 低 |
| Full descriptionの冒頭 | ★★★★☆ | 低 |
| アイコン | ★★★☆☆ | 高（再デザイン） |
| スクリーンショット2〜5枚目 | ★★★☆☆ | 高 |
| Full descriptionの中盤以降 | ★★☆☆☆ | 低 |
| プロモーションタイル（440×280） | ★★☆☆☆ | 中 |

**優先順位の結論**: Short descriptionの1行目→Full descriptionの冒頭→1枚目スクリーンショットの順に最適化する。変更コストが低い割に効果が大きい。

## ヒーロー情報の設計：最初の3秒で決まる

ユーザーがストアページを開いてから「インストール」ボタンを押すか離脱するかを決めるまで、平均3〜5秒です。この「ファーストビュー」に表示される要素——アイコン・タイトル・1枚目スクリーンショット・Short descriptionの最初の50文字——を「ヒーロー情報」と呼びます。

### Short descriptionの1行目設計

133文字の最初の50文字が勝負です。ここだけが「折りたたみ前」に表示されます。

```
# ❌ 機能説明型（弱い）
"ReadMark saves your scroll position on web pages."

# ✅ ベネフィット型（強い）
"Stop losing your place in long articles. ReadMark saves exactly where you left off."
```

**変更前後のReadMark CVRデータ:**
- 変更前（機能説明型）: CVR 73%
- 変更後（ベネフィット型）: CVR 89%

ユーザーは「何ができるか」ではなく「自分の何が解決されるか」を瞬時に判断しています。動詞の選び方が重要です。

```
# 弱い動詞（機能説明）
saves / stores / records / tracks / captures

# 強い動詞（ベネフィット表現）
Stop [問題] → [解決] で始める
Never [問題] again → [方法] で始める
Automatically [ユーザーが嬉しいこと] で始める
```

### アイコンの役割

アイコンはCVRへの直接影響は中程度ですが、「信頼感」に影響します。

**良いアイコンの条件:**
- **ひと目で機能が伝わる**: ReadMarkなら本にしおりを挟むアイコン
- **16×16でも識別できる**: タスクバーやブックマークバーでも見える
- **背景色は白か透明**: CWSの白背景と馴染む

**よくある失敗**: ブランド名の頭文字だけのアイコン（"R"や"P"）。ユーザーには「何の拡張か」が全く伝わらない。

## 詳細説明文の説得術：Problem/Features/Proof/CTA構造

Full descriptionのCVRへの影響は「ストアページを30秒以上見たユーザー」に集中します。離脱しなかったユーザーを説得するパートです。

### 4ブロック構造

```markdown
## [Block 1: Problem — 課題の共感（3〜5行）]
Long articles are frustrating to revisit. Every time you come back,
you have to scroll down to find where you left off — wasting minutes
that could be spent reading.

## [Block 2: Features — 機能の列挙（箇条書き）]
✨ Key Features
- **Auto-save**: Your position is saved automatically as you scroll
- **One-click restore**: Return to your exact spot instantly
- **Multi-tab support**: Track multiple articles simultaneously
- **Zero setup**: Works immediately — no accounts, no configuration

## [Block 3: Proof — 実績・信頼（具体的な数字）]
⭐ Trusted by 2,800+ readers
"Finally an extension that actually works on all sites." — ★★★★★

## [Block 4: CTA — 行動喚起（短く、明確に）]
📥 Install ReadMark — free, no account required.
```

### 箇条書きと段落のリズム

問題提起はあえて段落で書きます。共感を呼ぶには「読ませる」必要がある。機能列挙は箇条書きで。スキャンするユーザーに「この機能がある」と伝えるためです。

**アンチパターン**: 全て箇条書きにする。機能の羅列になり、「なぜ使うべきか」が伝わらない。

### 文章の長さの目安

```
Block 1 (Problem): 3〜5行（50〜100語）
Block 2 (Features): 箇条書き5〜8項目
Block 3 (Proof): 2〜3行（実績数字必須）
Block 4 (CTA): 1行
---
合計: 400〜600語（日本語なら800〜1,200字）
```

2,000字を超える説明文を書いている拡張が多いですが、後半は誰も読みません。簡潔さが説得力を生みます。

## スクリーンショット5枚の物語設計

スクリーンショットは「静止画のデモ動画」です。5枚を通してユーザーに「Before→After」の体験をさせます。

### 5枚構成の黄金パターン

| 枚数 | 役割 | 内容例（DataPickの場合） |
|-----|-----|---------------------|
| 1枚目 | **Hook（問題と解決の宣言）** | 「クリックするだけでデータを抽出」+ ハイライト選択UI |
| 2枚目 | **主要機能のデモ** | CSV/Excelエクスポート後のスプレッドシート画面 |
| 3枚目 | **差別化機能** | Google Sheets直接書き込みのデモ |
| 4枚目 | **ユースケース** | 対応サイト一覧（EC・不動産・求人等） |
| 5枚目 | **CTAまたは信頼構築** | フリーミアム境界（無料でできること/Proでできること） |

### 1枚目の5秒ルール

1枚目だけが検索結果サムネイルに表示されます。5秒見て何の拡張か伝わらなければ失格です。

**必須要素:**
1. **大きな英語キャッチコピー**（背景色とのコントラスト比 4.5:1以上）
2. **実際の動作画面**（モックアップより実画面の方が信頼される）
3. **機能名より「何が変わるか」**

```
❌ 弱い1枚目
"ReadMark - Save Reading Position"（機能の説明だけ）

✅ 強い1枚目
"Never lose your place again."（問題解決の宣言）
+ スクロール位置が保存されている実画面
```

**推奨サイズ**: 1280×800px、PNG、余白16px以上。小さすぎると検索サムネイルで潰れます。

## ソーシャルプルーフの配置

「他の人も使っている」という事実は、インストールの心理的障壁を下げます。

### 使えるソーシャルプルーフの種類

| 種類 | 効果 | 記載場所 |
|-----|-----|---------|
| インストール数（「2,800+」） | 需要の証明 | Short description / Full description Block 3 |
| 星評価（「★4.8 / 5」） | 品質の証明 | Full description Block 3 |
| ユーザーレビューの引用 | 感情的共感 | Full description Block 3 |
| メディア掲載（ProductHunt等） | 権威性 | Full description |

### 実際の記載例

```
# short-description の活用例
"Trusted by 2,800+ readers on any website. Save scroll position
with one click — restore it when you return. No account needed."

# full-description Block 3 の例
⭐ 2,800+ installs | ★4.8 average rating

"I've tried everything. This is the only extension that actually
works on every site I visit." — ★★★★★ (Chrome Web Store)
```

**注意**: インストール数が少ない段階（100未満）ではインストール数を強調しない。信頼より疑念を生む場合があります。代わりに「No account required」「Privacy-first」「Works on any site」等の機能的信頼要素を前面に出す。

## A/Bテスト指針：1回に1要素だけ変更する

CVRの改善はA/Bテストで検証します。ただし、CWSにはA/Bテスト機能がないため、「変更 → 測定期間 → 比較」を手動で行います。

### テスト手順

```
1. ベースライン測定
   → 変更前の2週間のCVR（インストール数÷ページビュー）を記録

2. 1要素だけ変更
   → Short descriptionの1行目だけ、または1枚目スクショだけ

3. 同じ期間を測定
   → 変更後の2週間のCVRを記録

4. 判定
   → 10%以上の改善 → 採用
   → 変化なし or 悪化 → 元に戻す
```

**最小サンプルサイズ**: 週100クリック未満の拡張では、2週間ではサンプルが不足します。この場合は4週間以上測定するか、「体感」より「方向性」を重視して意思決定します。

**変更してはいけない組み合わせ:**
- Short descriptionとFull descriptionを同時に変更
- 1枚目スクショとアイコンを同時に変更

因果関係が特定できなくなります。

## よくある失敗パターン5つ

18本を公開し、他の開発者のリスティングを分析してわかった「CVRを下げる典型パターン」です。

**① 機能羅列型の説明文**

```
❌ "Features: Save scroll position, Restore reading position,
Multi-tab support, Export settings, Dark mode, Keyboard shortcut..."
```

機能の箇条書きをひたすら並べる。ユーザーは「自分の問題が解決されるか」を知りたいのに、「何ができるか」だけを伝えている。必ずProblemから始める。

**② ベネフィット不在の1枚目スクリーンショット**

拡張のUI画面だけを貼り付けた1枚目。「何の拡張か」は伝わるが「なぜ使うべきか」が伝わらない。必ずキャッチコピーを画像上に重ねる。

**③ 過剰な機能列挙（20項目以上の箇条書き）**

多機能をアピールしたくなるのはわかる。しかし20項目以上の箇条書きは「読む気を失わせる」。本当に差別化できる3〜5機能だけを深く伝える方が効果的。

**④ 「最高」「業界最速」等の根拠のない修飾語**

CWSのポリシー違反になる可能性があるだけでなく、信頼を損ないます。具体的な数字（「2,800インストール」「34ms以下の応答時間」）の方が説得力があります。

**⑤ ターゲットユーザーの不明確化**

「誰でも使えます」は「誰にも刺さりません」。`Perfect for researchers, students, and anyone who reads long articles` のように、具体的なユーザー像を提示する。

---

:::message
**この章で学んだこと**

- CVRはタイトル最適化だけでは改善しない。SEOとリスティング最適化は別の施策
- Short descriptionの最初の50文字（ファーストビュー）が最もCVRへの影響が大きい
- Full descriptionはProblem/Features/Proof/CTAの4ブロック構造で書く
- スクリーンショット5枚は「課題提示→解決→詳細→ユースケース→CTA」の物語として設計する
- A/Bテストは1回に1要素のみ変更し、最低2週間測定する
:::
