[](#%E3%81%82%E3%82%8B%E6%97%A5%E7%AA%81%E7%84%B6%E3%80%81300%E4%B8%87%E4%BA%BA%E3%81%8C%E4%BD%BF%E3%81%A3%E3%81%A6%E3%81%84%E3%81%9F%E3%83%84%E3%83%BC%E3%83%AB%E3%81%8C%E6%B6%88%E3%81%88%E3%81%9F) ある日突然、300万人が使っていたツールが消えた

Web開発者なら一度は使ったことがあるだろう。**EditThisCookie**。Chromeでクッキーを確認・編集するための定番拡張機能で、300万人以上のユーザーがいた。


ところが2024年末、Googleの**Manifest V3移行ポリシー**により、MV2のまま放置されていたEditThisCookieはChrome Web Storeから削除された。


さらに問題なのが、残った「EditThisCookie V3」という拡張機能。これは元の開発者ではなく**別人がリライトしたもの**で、マルウェア混入が報告されるという事態まで起きた。


「cookie editor chrome」で検索すると出てくるのは、更新が止まったCookie-Editor（200万ユーザー、2024年2月以降更新なし）か、評価3.3のCookieManagerばかり。


**市場に空白ができていた。**


[](#%E3%80%8C%E4%BD%9C%E3%82%8B%E5%81%B4%E3%80%8D%E3%81%AB%E5%9B%9E%E3%81%A3%E3%81%9F%E7%90%86%E7%94%B1) 「作る側」に回った理由

僕は普段、Chrome拡張機能を個人開発している。S-Hubというブランドで17本ほど公開していて、ShortsKillerやX Detoxなど、日常の「ちょっとした不便」を解消するツールが多い。


クッキーマネージャーは正直、自分の得意領域からは外れる。使うのは主にWeb開発者やQAエンジニアで、僕の既存ユーザー層とは違う。


でも2つの理由で「作ろう」と決めた。


**① 市場ギャップが明確すぎる**

300万人の受け皿がない状態は、個人開発者にとって滅多にないチャンス。需要があるのは分かっている。問題は供給だけだ。


**② 開発者はお金を払う層**

僕のこれまでの拡張機能は無料ユーザーが多い。でもクッキー管理ツールのユーザーは開発者。開発者は良いツールに課金する文化がある。フリーミアムモデルとの相性がいい。


[](#cookiejar%E3%81%AE%E8%A8%AD%E8%A8%88%E6%80%9D%E6%83%B3) CookieJarの設計思想

名前は**CookieJar**にした。クッキーの瓶。シンプルで覚えやすい。


既存の競合を分析して見えてきたのは、**どれも「ただのクッキーエディタ」で止まっている**ということ。閲覧・編集・削除はできるけど、それ以上の価値を提供していない。


CookieJarでは「クッキーを見る」だけでなく、**「クッキーを理解する」**ところまで踏み込んだ。


[](#%E3%82%B9%E3%83%9E%E3%83%BC%E3%83%88%E8%87%AA%E5%8B%95%E5%88%86%E9%A1%9E) スマート自動分類

すべてのクッキーに自動でカテゴリバッジを付ける。


- 🟢 **Login** — セッション・認証系（`session`, `auth`, `csrf` 等）

- 🔴 **Tracking** — トラッキング系（`_ga`, `_fbp`, Disconnectリストのドメイン等）

- 🟡 **Analytics** — アナリティクス系（`_gid`, `_hjid` 等）

- 🔵 **Functional** — 機能系（`lang`, `theme`, `consent` 等）

- ⚪ **Unknown** — 上記に該当しないもの


分類ロジックはスコアリング方式で、名前パターンマッチ・ドメインマッチ・属性（HttpOnly, Secure, SameSite）を組み合わせて判定している。外部APIは使わない。完全にクライアントサイドで動く。


[](#%E3%83%97%E3%83%A9%E3%82%A4%E3%83%90%E3%82%B7%E3%83%BC%E3%82%B9%E3%82%B3%E3%82%A2) プライバシースコア

サイトを開くと、そのサイトのクッキーから0〜100のプライバシースコアを算出して表示する。トラッキングクッキーが多いサイトはスコアが低くなる。


開発者にとってはデバッグ情報、一般ユーザーにとっては「このサイトはどれくらい自分を追跡しているか」の直感的な指標になる。


[](#%E7%84%A1%E6%96%99%E3%81%A7%E5%8D%81%E5%88%86%E4%BD%BF%E3%81%88%E3%82%8B%E3%80%81%E3%81%A7%E3%82%82pro%E3%82%82%E3%81%82%E3%82%8B) 無料で十分使える、でもProもある

フリーミアムモデルを採用した。


**無料で使える機能：**


- クッキーの閲覧・編集・作成・削除

- 検索・フィルタ・ソート

- JSON / Netscapeエクスポート・インポート

- スマート自動分類

- プライバシースコア


**Pro（$4.99/月 or $29.99買い切り）：**


- 自動クリーンアップスケジュール

- localStorage / sessionStorage / IndexedDB の統合管理

- クッキープロファイル切替（開発/テスト/本番）

- 高度なエクスポート（cURL, Puppeteer, CSV）


基本機能を無料にしているのは、**EditThisCookieのユーザーをまず取り込む**ため。「前のツールと同じことができる」という安心感がないと、乗り換えてもらえない。Proは「これがあると嬉しいけど、なくても困らない」ラインを狙った。

MV3のService Worker制約は地味にハマるポイントが多い。


**やってはいけないこと：**


```
// ❌ グローバル変数に状態を保持（30秒で消える）
let cachedData = [];
```

**正しいやり方：**


```
// ✅ chrome.storage.localで永続化
const { data } = await chrome.storage.local.get('data');
```

Service Workerは非アクティブ状態が30秒続くと終了する。グローバル変数は全て消える。定期処理は`setInterval`ではなく`chrome.alarms`を使う。イベントリスナーはトップレベルに同期的に登録する。


この辺は知っていれば避けられるけど、知らないとデバッグで数時間溶ける罠。


[](#claude-code%E3%81%A7%E9%96%8B%E7%99%BA%E3%82%92%E9%AB%98%E9%80%9F%E5%8C%96%E3%81%97%E3%81%9F) Claude Codeで開発を高速化した

開発にはClaude Codeを使った。具体的には：


**Claudeで仕様書を作成** — 競合分析・機能設計・技術制約をまとめる

**仕様書を実装指示書に変換** — Phase分割＋サブエージェント構成

**Claude Codeに指示書を渡して実装** — Phase 1（MVP）→ Phase 2（分類＋Side Panel）→ Phase 3（Pro機能）


サブエージェントは「並列で作れる独立したモジュール」にだけ使った。たとえばPhase 1では：


**サブエージェントA**: クッキー分類エンジン（background）

**サブエージェントB**: Popup UIコンポーネント群

**サブエージェントC**: Background Service Worker


型定義（`types.ts`）はサブエージェントに任せず、メインで先に作る。ここをサブエージェントにすると型の不整合が起きるリスクがある。


ポイントは**「仕様書をそのまま渡さない」**こと。仕様書は「何を作るか」、指示書は「どう作るか」。Claude Codeが必要なのは後者。


[](#cws%E6%8F%90%E5%87%BA%E3%81%A7%E6%B0%97%E3%82%92%E3%81%A4%E3%81%91%E3%81%9F%E3%81%93%E3%81%A8) CWS提出で気をつけたこと

Chrome Web Storeの審査で過去にリジェクトされた経験がある。AIサービス名を列挙したら「キーワードスパム」で弾かれた。


今回学んだCWSのルール：


**同一キーワードの繰り返しは5回まで**（cookieという単語の使用回数を数える）

**競合拡張機能名は絶対に書かない**（「EditThisCookieの代替」はNG）

- **「best」「fastest」等の誇大表現は禁止**

- **ブランド名の列挙は5つまで**


タイトルは `CookieJar - Cookie Editor & Manager` にした。CWS検索で最重要の「cookie editor」と「cookie manager」の両方をカバーしつつ、75文字制限に余裕を持たせた構成。


[](#%E3%82%84%E3%81%A3%E3%81%A6%E3%81%BF%E3%81%A6%E6%80%9D%E3%81%A3%E3%81%9F%E3%81%93%E3%81%A8) やってみて思ったこと

正直、クッキーマネージャーは**自分が毎日使うツールではない**。ShortsKillerやX Detoxは自分自身がヘビーユーザーだったから作ったけど、CookieJarは「市場にチャンスがあるから」作った。


これは個人開発において結構重要な転換点だと思っている。


「自分が欲しいものを作る」フェーズから、「需要があるものを作る」フェーズへ。


両方のアプローチに正解はない。でも収益化を考えると、後者の視点は避けて通れない。自分が使わないツールでも、ユーザーの声を聞いて改善していけば、良いプロダクトにはなる。
