18本のChrome拡張を開発して、売れる法則が見えてきました。

すべてうまくいったわけではありません。18本のうち数本は、今もひっそりと12インストールで眠っています。でも、伸びた拡張と伸びなかった拡張の差は、私が最初に思っていたものとは全然違うところにありました。

## 個人開発の孤独と、書籍化を決めた理由

個人開発を始めたのは、会社の業務で使いたいツールが見当たらなかったのがきっかけでした。「Chrome拡張なら週末で作れるんじゃないか」と思って、Claudeを使いながら最初の1本を書いたのが18ヶ月前です。

最初の1週間で40インストール。同僚とRedditからです。

でも、その後が思ったより静かでした。2本目、3本目を出しても、インストール数はなかなか伸びない。「なぜ伸びないのか」が分からないまま、試行錯誤を繰り返す日々が続きました。

当時、日本語で「Chrome拡張の収益化」について書かれた実践的な情報がほとんどなかったことを覚えています。英語の記事はあるものの、Chrome Web Storeのアルゴリズム変化や、日本語市場特有の動きには触れていない。

6本目で初めて有料ユーザーが生まれ、12本目でようやく毎月の収益が安定してきました。そのころから「ここまでの試行錯誤を、誰かが最初から読めるようにしたい」という気持ちが強くなりました。

書籍化したのは、情報を「整理して伝える」ためです。Zennの記事は断片的になりがち。本という形にすることで、順序立てて学べる構造を作れます。

## 18本作る中で気づいた5つのパターン

### Pattern 1: 「小さく始めて、継続する」は本当だった

最初の拡張に半年かけていたら、おそらく今も1本目を作っていたと思います。

1本目は週末で動くものを出す。2本目でMV3の設計を学ぶ。3本目でTypeScriptを導入する。4本目で収益化の仕組みを試す——このサイクルで進んだことが、結果的に最も速い学習経路でした。

「完璧に設計してから始める」よりも「動くものを出してフィードバックをもらう」サイクルの方が、圧倒的に多くのことを教えてくれます。Chrome拡張は、このサイクルを回すのに特に適した開発領域です。

### Pattern 2: タイトルを変えただけでインストールが340%増えた

ある拡張が2ヶ月間、週あたり約80インストールで横ばいでした。機能は変えていない。ただ、Chrome Web StoreのタイトルとKWIDを見直して、主要な検索キーワードをタイトルの最初の30文字に入れ直しました。

3週間後、週352インストールになりました。

Chrome Web Storeには検索アルゴリズムがあります。多くの開発者はCWSを「ZIPファイルをアップロードする場所」だと思っています。実際には、タイトル・ショートディスクリプション冒頭132文字・カテゴリ設定が検索順位に直接影響します。

私が見てきた個人開発者の多くのリスティングは、このあたりが最適化されていません。製品が良くても、見つかる仕組みを作らないと使ってもらえない——これが最初に気づいた大きな教訓です。

### Pattern 3: フリーミアムの「境界線」を引き直したらCVRが1.9%から3.2%になった

フリーミアム設計で最初に試したのは、「できるだけ早く制限に当たらせる」アプローチでした。ユーザーが価値を感じる前にペイウォールを見せれば、課金への動線が短くなるという考えです。

結果は逆でした。

制限に当たってから有料転換するユーザーの割合（CVR）は1.9%。無料範囲を広げた設計では3.2%になりました。

理由は使ってみると納得できます。ユーザーが「この拡張がないと困る」と感じる前にペイウォールを見せても、課金の動機がない。ペイウォールに当たる前に拡張が生活の一部になっていれば、「このまま使い続けたい」という気持ちが課金を後押しします。

今は「無料で使い続けられるが、ヘビーユーザーは自然に制限に当たる」という設計を基本にしています。

### Pattern 4: クロスプロモーションで1拡張のLTVが実質2倍に近づいた

複数の拡張を持っていると、既存ユーザーへの配信チャンネルができます。

私は使用開始から7日後に、他の自作拡張を小さなバナーで紹介するようにしました。既存ユーザーからの転換率は、CWS検索経由の新規ユーザーの約8倍です。

一度信頼を得たユーザーは、同じ開発者の別の拡張も試してくれやすい。このシンプルな事実が、拡張を増やすほど全体のLTVが上がる仕組みを作ってくれました。

1本目を作っている方にはまだ関係のない話ですが、2本目・3本目を考えているなら、早めに意識しておくと良い設計パターンです。

### Pattern 5: AIとのペア開発で生産性は約3倍になった——ただし使い方が重要

ClaudeとCursorを使った開発に切り替えてから、同じ機能を実装するのにかかる時間が大きく変わりました。

ただし、「AIに丸投げすれば速い」という話ではありません。

AIが得意なのは、ボイラープレートの生成、APIの使い方の説明、テストコードの土台作り、仕様の第一稿を実装に落とす作業です。

苦手なのは、背景情報なしでの設計判断です。「拡張を作って」とだけ伝えると、それらしく見えるが微妙にバグのあるコードが返ってきます。

私のルールは「AIが書いたコードは、自分で説明できるまで使わない」です。説明できないということは、理解していないということ。理解していないコードをマージすると、後でデバッグに何倍もの時間がかかります。

生産性3倍の実感は、「自分が理解していることをAIに速く実装させる」場面での数字です。設計判断は自分でする、実装をAIに手伝ってもらう——この役割分担が最もうまく機能しています。

## Book 1を無料にした理由

正直に言うと、経済的な判断でもあります。無料のBook 1を読んだ方が、Book 2・3の読者になってくれる可能性があるからです。

でも、それだけではありません。

私がChrome拡張を始められたのは、誰かが書いた無料の記事や、オープンソースのサンプルコードのおかげです。情報の恩恵を受けた側として、入り口の情報を有料にすることに違和感がありました。

「Chrome拡張を作ってみたいけど、何から始めていいか分からない」という状態で止まっている方が、日本語コミュニティにも一定数いると感じています。その方が最初の1本を作れるようになるためのコストを、私が少し負担できるなら、それは悪くない使い方だと思っています。
